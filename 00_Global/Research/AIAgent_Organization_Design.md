# AIエージェント組織設計 — 空気デザイン式運用ガイド
*出典: 「シリコンバレー17年のプロが本気でAIエージェント使い込んだ結果全部見せます」*
*チャンネル: 空気デザイン（りお） / 保存日: 2026-05-18*

---

## 核心メッセージ

> **「どのAIが強いかで議論するのは素人のゲーム。プロはどう回すかで勝負を決める。」**

AIエージェントの運用は「腕の差」ではなく **「運用設計思想の差」** である。
1対1でチャットするのと、組織として回すのでは、アウトプットの質・量に決定的な差が生まれる。

---

## 1. なぜ単体AIには天井があるか

### 問題①：出力がブレる
プロジェクトX、クライアントY、議事録、買い物リストを同じセッションに混ぜると、AIが迷って出力が歪む。

### 問題②：ハルシネーションが増える
文脈が散らかると「先週こう決まりましたよね」と決まっていないことを自信満々に言い始める。

### 問題③：品質チェックする人がいない
AIが一つだと、出力をレビューするのは自分だけ。これはスケールしない。
プロの現場では **設計・コード・ローンチに必ず複数の目が入る仕組み** がある。

## 2. シリコンバレーの組織原則をAIに適用する

17年（Google・Amazon・Cisco・スタートアップ）で学んだ「うまく回っている組織」の4要素：

| 原則 | 人間の組織 | AIエージェント組織への適用 |
|---|---|---|
| **専門化** | PM / エンジニア / デザイナー / SRE と役割を細分化 | 専門化したエージェントを役割ごとに設計する |
| **レビュー文化** | デザインレビュー・コードレビュー・ローンチレビュー | AIの出力を別のAIが多段階でレビューする |
| **書き物で引き継ぐ** | PRD・RFC・ADR・設計ドキュメントで意思決定を記録 | ファイル・タスクボードに書き込み、非同期でログを残す |
| **責任の所在の明示** | 誰がどこまで決めるかを明文化（担当→マネージャー→ディレクター） | ここまではAIが判断していい、ここから先は人間が判断するを明確化 |

> これらは人間が数千年かけてたどり着いた解決策。AIエージェントに当てはめればいい。

## 3. 空気デザインのAI組織構成

### 全体像

```
          【りお（CEO / 設計者）】
                   |
         ┌────────────────┐
         │    調整層（イントラネット）    │
         │  タスク管理 / Obsidian文脈エンジン  │
         └────────────────┘
                   |
    ┌──────┬──────┬──────┐
 エンジニア  コンテンツ  ビジネス  インフラ運用
  チーム    チーム    チーム    チーム
```

- **合計: 4チーム × 18体のAIエージェント**
- メインモデル: Claude（Sonnet / Opus / Haiku を役割で使い分け）

---

### AI社員の3要素（全エージェント共通）

| 要素 | 内容 | 例 |
|---|---|---|
| **マニュアル** | 何をするか定義した指示書（Skills.md） | 「あなたはUIデザイナーです。こういう責務を持ちます」 |
| **頭脳** | LLMモデル本体 | Claude Sonnet / Haiku / Opus / Gemini / Hermes |
| **机** | コンテキストを分離するセッション/サブエージェント | 文脈が混ざらないための独立した作業空間 |

---

### エンジニアリングチーム

```
テックリード（Sonnet / オーケストレーター）
  ├── QA（Haiku）
  ├── フロントエンドエンジニア（Haiku/Sonnet）
  └── バックエンドエンジニア（Haiku/Sonnet）

エンジニアリングディレクター（Opus / アドバイザー）
  └── 難しい判断・戦略的タスクのときだけ登場
```

**コスト設計のポイント:**
- Opus（最上位）は「戦略判断が必要なときだけ」呼ぶ
- 日常タスクはSonnet・Haiku で回してトークン節約
- テックリードがタスクの難易度を判断して適切なモデルに振る

---

### コンテンツチーム

```
コンテンツディレクター（オーケストレーター）
  └── 台本を膨らませる・バランス調整
       ↓
  ├── ブランドボイスチェッカー（空気デザインの価値観・口調と一致しているか）
  ├── ルート構図チェッカー（本質を突き詰めているか・賞味期限の長い情報か）
  └── アンチAIスロップ（AIっぽい不自然な表現・絶対にしゃべらない内容を除去）
```

---

### ビジネスチーム

```
マーケティングディレクター
ビジネスストラテジスト
  └── 数日おきにブリーフィングを出力

パートナーシップマネージャー
  └── 案件の事前フィルタリング（ハイプ製品・ユニークでない製品をはじく）

リーガルレビュー
  └── 契約書の第一次チェック（最終は人間の弁護士）
```

---

### インフラ運用チーム（最重要）

```
ローカルサポート（Hermes Agent / ローカルAI）
  └── タスク管理・Obsidianのゴミ掃除を日夜実行

タスクディスパッチャー
  └── Obsidianのインボックスを読み込み → タスクボードにタスク作成 → 適切な担当者にアサイン
```

## 4. Obsidian 文脈エンジンの構造

> AIと人間が共通で使う「会社のイントラネット」として機能。

### フォルダ構成

```
Obsidian Vault/
├── Inbox/          ← 全入力の受け口。AIが自動振り分け
├── Projects/       ← 進行中プロジェクト
├── Ideas/          ← アイデアストック
├── Resources/      ← 一時資料
└── Context/        ← 最重要。会社と自分に関する全文脈
    ├── Philosophy & Values     （哲学・価値観）
    ├── Professional Identity   （職務履歴・現在の立場）
    ├── Technical Setup         （使用ツール・機材）
    ├── Visual Design System    （デザイン言語・ブランド定義）
    └── AI Handoff/             ← AIエージェント間の引き継ぎ書置き場
```

### 自動振り分けフロー

```
スマホからメモ投下
      ↓
Inbox に着地
      ↓
タスクディスパッチャーが読み込み
      ↓
Projects / Ideas / Resources / Context に自動分類
      ↓
必要ならタスクボードにタスク追加
```

## 5. タスク管理ボード（人間とAI共通）

vibe コードで自作したカンバンボード。全AIと人間が同じボードを参照する。

### カラム構成

| カラム | 意味 |
|---|---|
| **Unassigned** | 未アサインのタスク。AIと人間がここから拾う |
| **In Progress** | 実行中 |
| **Blocked** | AIが詰まって人間の判断が必要 |
| **Done** | 完了 |

### タスクの流れ

```
りお / AI がタスクを Unassigned に追加
         ↓
タスクディスパッチャーが担当者を決定
         ↓
各エージェントが夜中に自律実行
         ↓
詰まったら Blocked + コメントで人間に差し戻し
         ↓
朝のモーニングスタンドアップで夜間実行結果を確認
```

## 6. 1000時間運用で得た3つの教訓

### 教訓①：最大の見返りは「時間節約」ではない
夜中のQA・セキュリティチェック・ドキュメント整理は、人間一人では割に合わない仕事。
組織化すると **「一人では絶対できなかった仕事」が経済的に成立する。**
これが時間節約とは桁違いのレバレッジ。

### 教訓②：AIに暗黙の了解は通用しない
> 「消すなと書いてなければ消す可能性がある」

実際に Obsidian のインボックスをAIが自判断で削除する事故が発生した。
**やっていいこと・いけないこと・迷ったときの判断基準を全部書き下ろす必要がある。**
組織が変化するたびに指示書を更新することも必須。

### 教訓③：AIの組織はエラーなく静かに壊れる
人間なら「このメッセージ届いていないのですが」とエスカレートしてくる。
AIは壊れたファイルに「書き込み完了しました」と返してから次の仕事に移る。
→ **定期的に端から端まで観察する健全性チェックの仕組みが必要。**

## 7. 3原則まとめ

```
① 一つのAIに全部やらせない
   役割を分けて、それぞれが自分の領域だけ責任を持つ
   文脈を分ける。品質チェックは複数の目で。

② 全員が同じ記憶を共有する
   Obsidian文脈エンジン ＋ 共有タスクボード = 共有メモリ
   「毎朝全員が記憶喪失で出社する会社」は成立しない

③ 引き継ぎはドキュメントで（会話ではなく）
   AIエージェント同士は直接会話しない
   タスクボードに書いて非同期でログを残す
   後から誰でもたどれる非同期コミュニケーション設計
```

## 8. 実践チェックリスト

- [ ] 各AIエージェントに専門の役割と指示書（マニュアル）を定義する
- [ ] コンテキストを分けるためのサブエージェント/セッション設計をする
- [ ] 出力をレビューする別のAIエージェントを用意する（1段階以上）
- [ ] Obsidianなどで「全エージェント共通の文脈エンジン」を作る
- [ ] AIと人間が同じタスクボードを使えるようにする
- [ ] タスクディスパッチャーを作り、自動でタスクを割り振る
- [ ] 「AIがやっていいこと / 人間が判断すること」の線引きを明文化する
- [ ] 夜間の自動QA・健全性チェックの仕組みを作る
- [ ] 定期的に全エージェントの動作を端から端まで観察するルーティンを作る
